## 修复清单

### P0/P1——应尽快修

#### 1. HttpPool::max_size 不是硬上限

**文件：** `src/http/http_pool.hpp`

**问题：** `max_size` 的 per-shard 检查是个空 if 块（只有注释），判断后没有 `return`，total/in_flight 仍然递增，因此 `max_size` 不是文档声称的硬上限。

**修复：**
- 改为轮询 `kShards` 个 shard，全部满则 `co_return nullptr` + 日志 WARN
- `max_concurrent` 仍 per-shard 立即返回（不改动）
- 除法从 `floor + 1` 改为 `(val + kShards - 1) / kShards`（ceil 除法），避免 +1 导致的全局上限溢出

```cpp
// 修复前：空 if 块
if (try_shard.total >= state->cfg.max_size / kShards + 1) {
    // Per-shard limit reached; could try another shard
}

// 修复后：轮询所有 shard
if (try_shard.total >= (state->cfg.max_size + kShards - 1) / kShards) {
    if (try_idx + 1 < kShards) {
        shard_idx = (shard_idx + 1) % kShards;
        continue;
    }
    all_shards_full = true;
    break;
}
```

#### 2. Chunked 请求转发错误

**文件：** `src/http/http_server.hpp`

**问题：** `read_chunked_stream()` 把 chunk body de-chunk 后写入 `ctx.body`，但转发时仍保留 `Transfer-Encoding: chunked`。上游收到 `Transfer-Encoding: chunked` 但 body 不是 chunked framing，协议错乱。

**修复：**
- 转发时跳过 `Transfer-Encoding`（如果原请求是 chunked）
- 写 `Content-Length: <dechunked-size>` 替代
- 空 body 也写 `Content-Length: 0`（去掉 `&& !ctx.body.empty()` 条件）

```cpp
// 修复前
if (request_header_state.is_chunked && !ctx.body.empty()) {
    forward_req += "Content-Length: " + std::to_string(ctx.body.size()) + "\r\n";
}

// 修复后
if (request_header_state.is_chunked) {
    forward_req += "Content-Length: " + std::to_string(ctx.body.size()) + "\r\n";
}
```

#### 3. normalize_path() 没剥离 query string

**文件：** `src/security/path_normalize.hpp`

**问题：** `normalize_path()` 处理完整 request-target，`/api/health?x=1` 不会命中白名单 `/api/health`。黑名单和限流的 path 匹配也被 query 干扰。

**修复：** 在函数开头按 `?` 分割，只规范化 path 部分。

```cpp
// 修复前：直接处理 full path_only
inline NormalizedPath normalize_path(std::string_view path_only, bool case_sensitive = false) {
    std::string decoded = percent_decode(path_only);

// 修复后：先剥离 query
inline NormalizedPath normalize_path(std::string_view path_only, bool case_sensitive = false) {
    auto qpos = path_only.find('?');
    if (qpos != std::string_view::npos) {
        path_only = path_only.substr(0, qpos);
    }
    std::string decoded = percent_decode(path_only);
```

## 压测结果说明

### 可信的部分

- **功能未回退：** 三向日志闭环验证。压测期间客户端 socket error = 0，上游（zebra-config）日志 0 error / 0 timeout / 0 cancel，网关自身日志 0 error / 0 warn / 0 timeout。功能正确性经得起检验。
- **连接池未触顶：** HttpPool 在全部 shard 满时会输出 `LOG_WARN`。整轮压测 0 条 WARN，说明连接池从未被打满。
- **复用正常工作：** 上次 tcpdump 验证的 99%+ 复用率结论仍然有效，本次未改动连接池逻辑。

### 需要注意事项

本次压测为 **同机烟雾测试（功能验证用），不代表性能上限**。以下因素限制了数字的可信度：

1. **同机竞争：** wrk 和 server 都在同一台 VM，抢占 CPU。30 线程 wrk + server 多线程争抢 6 核，导致 RPS 上限不是 server 的真实吞吐。
2. **2 轮不足：** 没有 warmup 轮，缺少分位数统计。
3. **30s 时长偏短：** 不足以暴露池抖动或内存碎片。
4. **环境信息缺失：** 未披露 vCPU 数、pool_size 配置、上游规格等。
5. **网关效率 50.3% 不可信：** 直连 18K vs 网关 9K 的差距更可能是 wrk 同机自挤造成的，不代表真实网关 overhead。

### 后续改进方向

待跨机压测资源到位时，建议：

1. **跨机压测：** 换一台 VM 运行 wrk，消除同机 CPU 竞争。
2. **wrk 加 `--latency`：** 获取 p50/p90/p99 尾部分布。
3. **3+ 轮（含 warmup）：** 丢弃第一轮，取均值 ± stddev。
4. **5min 长压：** 暴露池抖动和内存碎片。
5. **补环境清单：** nproc、free -h、pool_size 配置、上游规格。
6. **perf / pprof：** 抓 server 自身热点，比 RPS 数字更有信息量。

等跨机数据出来后再替换本次结果。
