## 一、核心 Bug 修复

### 根因

`src/http/http_pool.hpp` 的 `is_reusable_idle()` 中，`non_blocking()` 恢复调用覆盖了 `receive()` 的 `ec`：

```cpp
// 修复前：ec 被覆盖，would_block 丢失
size_t n = conn.socket.receive(asio::buffer(&byte, 1), asio::socket_base::message_peek, ec);
conn.socket.non_blocking(was_non_blocking, ec);  // ← 覆盖了 ec

// 修复后：recv_ec / restore_ec 分离
asio::error_code recv_ec;
size_t n = conn.socket.receive(asio::buffer(&byte, 1), asio::socket_base::message_peek, recv_ec);
asio::error_code restore_ec;
conn.socket.non_blocking(was_non_blocking, restore_ec);
```

健康 idle keep-alive 连接上，`peek` 没有数据可读时返回 `would_block`，这是正常状态，表示连接可复用。旧代码把 `would_block` 覆盖成 success + `n=0`，导致所有 idle 连接都被判定为不可用。

### 影响

修复前 HttpPool stats 显示 `reused=0`、`created=请求数`、`probe_dropped≈请求数`——**每条请求都新建 TCP 连接，无复用**。tcpdump 也确认 stream 数 = 请求数。

## 四、数据一致性说明

本轮排查中发现了之前 tcpdump 分析方法的错误：

| 方法 | 问题 | 正确做法 |
|:-----|:------|:---------|
| `-c 1000` + SYN-ACK 比例估复用率 | 样本太小，SYN-ACK 占比不反映复用率 | 应使用全量 pcap + `tshark -e tcp.stream` |
| 小样本间接推断 | 两次误判为"复用正常" | 直接用 `tcp.stream` 计数 |

修复后，`tcp.stream` 与 HttpPool stats 双双证明复用正常，结论可靠。
