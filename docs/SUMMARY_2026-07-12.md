## 一、修复清单

### 第一阶段：HttpPool 连接复用 + HTTP 层修复

| # | 问题 | 根因 | 修复 |
|:--|:-----|:-----|:------|
| C3 | `reused=0`，连接从未复用 | `is_reusable_idle` 中 `non_blocking` 恢复覆盖了 `recv` 的 `ec`，`would_block` 被误判为不可用 | `recv_ec`/`restore_ec` 分离 |
| H1 | 代理响应 Content-Length 不匹配 | `json_keys_snake_to_camel` 变换后 body 大小变了，但上游 CL 被透传 | 有 body 响应过滤上游 CL，HEAD 响应保留 |
| H3 | accept 失败无退避 | `start()` 中 catch 后立即重试 | 指数退避 10ms-1000ms |
| H4 | 代理请求 header 无 CRLF/NUL 注入检查 | `build_proxy_request` 直接转发 header | 检测后返回空字符串，调用方返回 400 |
| S3 | max_concurrent 限流在循环内误判 | while 循环内第一个 shard 失败就 return | 移到 acquire 入口单次检查 |
| S4 | 超时后未 cancel | `parallel_group::wait_for_one` 后异步操作未显式取消 | 超时分支加 `cancel()` |
| S5 | 惰性回收 | `evict_stale_idle` 仅在 acquire 时调用 | 新增公开 `evict_stale()` 方法 |

### 第二阶段：MySQL 连接崩溃排查

| 尝试 | 方案 | 结果 |
|:----:|:-----|:-----|
| 1 | `mysql_thread_init` 补全 | ❌ 仍然 crash |
| 2 | `asio::post` 参数顺序修正 | ❌ 编译不支持 |
| 3 | `unique_ptr<string>` lambda 捕获 | ❌ heap-use-after-free（ASan 定位） |
| 4 | `shared_ptr<string>` lambda 捕获 | ❌ heap-use-after-free（ASan 定位） |
| 5 | `mysql_result_to_json` stub | ❌ 没跑到那里就崩了 |
| 6 | **`co_await asio::post(executor, use_awaitable)` 仅切换 executor** | ✅ **修好** |

### 第三阶段：最终修复

**根因（GCC 11 协程 bug）：** `co_await asio::post(lambda, executor, use_awaitable)` 中，GCC 11 对临时 lambda 闭包对象用 `memcpy` 复制，不走 copy constructor。导致 `unique_ptr`、`shared_ptr`、`string` 等 lambda 捕获对象被"浅拷贝"，内部指针共享同一块内存，在不同线程（worker T2 + io_context T36）各析构一次 → heap-use-after-free。

**最终方案（`mysql_pool.hpp:117-122`）：**

```cpp
// 修复前（所有 lambda 捕获方案均失败）
auto sql_ptr = std::make_unique<std::string>(std::move(sql));
Result res = co_await asio::post(
    [this, sql_ptr = std::move(sql_ptr)]() -> Result { return do_query(sql_ptr->c_str()); },
    worker_pool_, asio::use_awaitable);
co_return res;

// 修复后（仅切换 executor，无 lambda 捕获）
co_await asio::post(worker_pool_, asio::use_awaitable);  // 切到 worker 线程
if (!running_) { co_return Result{false, "mysql pool stopped", ""}; }
co_return do_query(sql.c_str());  // sql 在协程帧内，单一所有权
```

**ASan 验证：** 3 次精准定位（PID 6073/9281/11395），最终修复后 `heap-use-after-free` 消失，MySQL 返回 HTTP 200。

#### 场景 B：上游正常（最终数据，gRPC 后端 ~15.9k RPS）

| 指标 | Direct | Gateway | 损耗 |
|:-----|:------:|:-------:|:----:|

| avg_lat | 6.5ms | 7.6ms | +1.1ms |

**29% 是网关代理的真实损耗，不是 bug。** 每个网关请求比直连多出：

1. **下游 HTTP 解析**（picohttpparser 解析请求头）
2. **HttpPool acquire/release**（分片锁 + idle 队列操作）
3. **build_proxy_request**（重建 HTTP 请求头、header 过滤/校验）
4. **上游 write + read**（与直连相同的网络 IO）
5. **read_proxy_response**（解析上游响应头 + body）
6. **json_keys_snake_to_camel**（逐字符扫描 JSON，蛇形→驼峰键名转换）
7. **build_downstream_response**（重建 HTTP 响应头、过滤 hop-by-hop 头）

以上全部跑在同一个 `asio::io_context` 上。单次请求直连只需步骤 4，网关需要全部 7 步。

## 五、关键教训

1. **GCC 11 协程 + `co_await asio::post(lambda)` 是危险组合。** 所有 lambda 捕获（move、copy、unique_ptr、shared_ptr）在 GCC 11 下都可能触发 memcpy 浅拷贝导致 double free / use-after-free。唯一安全的跨线程方案是 `co_await asio::post(executor, use_awaitable)` 仅切换 executor，数据留在协程帧内。

2. **tcpdump 小样本 SYN-ACK 比例不能用来估算复用率。** 必须用全量 pcap + `tshark -e tcp.stream` 按流统计 HTTP 请求数。

3. **ASan + `-g -fno-omit-frame-pointer` + `symbolize=1` 是定位内存 bug 最有效的工具组合。** 3 次精准定位，每次 10 分钟从构建到结果。
